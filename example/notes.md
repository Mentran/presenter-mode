# Demo speaker notes

## 01 The projection trap

这一页要讲清楚一个真实痛点：HTML PPT 本身没有错，问题出现在投屏之后。

观众看到的是投影屏，演讲者自己的屏幕往往也被展示窗口占满。这个时候，原来放在旁边的讲稿、临时提醒、下一页预览都没了。

这里可以举一个很具体的场景：页面上只有一句话，但你真正要讲的是这个观点的背景、为什么重要、一个例子，以及如何过渡到下一页。

重点不是“我需要一个提词器”，而是“我需要一个不会暴露给观众的演讲控制台”。

转场：所以，越是高级、克制、信息少的 PPT，越依赖演讲者背后的讲述系统。

## 02 Light slides, heavy talk track

这页要反直觉一点：很多人以为页面简单，演讲就简单。其实往往相反。

页面简单，是为了让观众把注意力放在你正在讲的内容上，而不是在屏幕上读小字。观众看到的是结论，演讲者要负责把结论展开。

可以这样讲：

- PPT 页面承担的是视觉锚点
- 讲稿承担的是逻辑链条
- 演讲者承担的是节奏和现场判断

如果没有 Presenter Mode，你会把这三件事混在一起处理。结果就是：要么把 PPT 做得太满，要么现场只能靠记忆硬撑。

转场：当讲稿和页面分离后，如果没有专门的演讲者视图，代价会很快出现。

## 03 What goes wrong without notes

这一页用来放大痛感，不要讲功能，讲后果。

第一个代价是忘掉转场。很多卡顿不是因为忘了这一页的内容，而是不知道下一句话该怎么接。尤其 AI、产品、技术类分享，逻辑跳跃很容易让观众掉线。

第二个代价是节奏失控。没有计时和下一页预览时，演讲者很难判断“这里还能不能多讲一个例子”。最后常见结果是前面太细，后面赶时间。

第三个代价是操作分心。切窗口、找讲稿、确认页码，这些动作单个看都很小，但在演讲现场会持续消耗注意力。

这里要强调：Presenter Mode 解决的不是“翻页”，而是减少临场认知负担。

转场：所以正确的结构不是让一个窗口承担所有事情，而是把观众视图和演讲者视图拆开。

## 04 Presenter Mode split

这一页讲产品思路：两个窗口，两个角色。

观众窗口只负责展示 PPT。它应该干净、稳定、适合投影，全屏后不要出现任何讲稿、控制按钮或调试信息。

演讲者窗口则是私有控制台。它应该能看到当前页、下一页、讲稿、计时、页码列表，并且能控制观众窗口同步翻页。

这里可以顺带解释为什么不是“在 PPT 页面上加备注层”：因为备注层一旦和观众视图耦合，就会带来投屏泄露风险，也会让页面代码更复杂。

Presenter Mode 的目标是作为旁路工具存在：尽量少侵入原来的 PPT，只要求一个很小的同步协议。

转场：那这个协议应该小到什么程度？

## 05 Simple contract

这一页讲接入规范，重点是“低门槛”。

默认情况下，新用户只需要把 PPT 放在 `slides.html`，讲稿放在 `notes.md`，然后打开 `presenter.html`。这适合最简单的项目。

如果用户已经有复杂目录，也不应该强迫改名。所以支持 URL 参数：

`presenter.html?slides=deck/index.html&notes=talk/script.md`

PPT 侧最低要求是每页有 `.slide`。如果 PPT 提供 `window.deck.show(index)`，Presenter Mode 就调用它；如果没有，就退回到 hash：`#1`、`#2`。

讲稿侧用 `## 01 Title`、`## 02 Title` 对齐页码。这比把讲稿塞进 HTML 注释里更容易写，也更容易维护。

转场：这也决定了这个项目的边界。

## 06 Open source direction

最后一页要讲开源定位：不要把它讲成大而全的演示平台。

它不是 PPT 编辑器，不负责帮你设计页面，不负责生成内容，也不需要接管所有框架。

它应该专注在演讲现场最需要的几件事：

- 让观众窗口稳定展示
- 让演讲者窗口可读、可控、可折叠
- 让路径加载和页码同步足够可靠
- 让新用户 3 分钟内完成第一次接入

结尾可以说：开源初版最重要的不是功能多，而是“拿来就能解决一个明确痛点”。如果这个 demo 能让用户一眼理解为什么需要 Presenter Mode，它就是合格的。
